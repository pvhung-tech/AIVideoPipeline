# Phase 10 Release Notes

Status: Release validation started; first-release startup gate revised

Date: 2026-07-14

## Release Gate Decision

For the first release candidate, Phase 10 uses the revised installed-app
startup gate:

- NSIS installed-app startup average must be `<= 6s`.
- Every startup attempt must return a healthy `/api/health` response.
- Installed-app smoke must discover the installed desktop executable and
  sidecar from the installation directory.
- Temporary install cleanup must complete.
- First-run packaged one-file cold start remains tracked separately.

The previous strict `max <= 5s` target remains the long-term release hardening
goal. It is moved out of the first release blocker list because the remaining
gap is dominated by PyInstaller one-file cold-start behavior and Windows
process/cache variance rather than application readiness after installation.
The next technical spike for that target is a non-one-file or onedir sidecar
packaging path.

## Notable Changes In This Candidate

- Phase 9 Optimization is closed.
- Render performance has official deterministic and live-provider baselines.
- Media workspace supports provider partial results for slow or rate-limited providers.
- Cached media panel uses a 100-item window so 2500-entry cache refresh first paint stays below `0.5s`.
- Installers were rebuilt with the refreshed sidecar and frontend bundle.
- Desktop launch no longer depends on opening the app data log file. The Tauri
  log plugin now writes to stdout only, preventing stale log-file permission
  failures from blocking app startup.
- FastAPI sidecar cold-start imports were reduced by lazy-loading app/server
  setup in `sidecar.py`, pinning the desktop Uvicorn runtime to local HTTP
  `h11`, and deferring heavy provider/service imports out of FastAPI router
  module import time.

## Known Release Risk

Packaged FastAPI sidecar startup still exceeds the long-term strict 5-second
max target on this machine, although the optimized installed-app smoke passes
the revised first-release gate and consistently returns healthy responses.

The remaining gap is likely PyInstaller one-file cold-start and Windows
process/cache variance. Further meaningful reduction probably requires a
packaging strategy decision such as onedir-style sidecar deployment or accepting
a slightly higher first-release startup target.
